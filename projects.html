search

this project was written in scala. given an XML file containing pages that link to one another, it will index the pages and then query the results, similar to how modern search engines work. additionally, it implements Google's PageRank algorithm, which accounts for page linkages in search results.

usage

Before any querying can begin, the indexer must be run. It takes exactly one argument, which is the filepath that leads to a valid XML file. This program will write three files--titles.txt, docs.txt, and words.txt--all of which will be used to query the document. Once the indexer has completed, you will be able to query that file via the querier.

The querier takes in either three or four arguments. If you would like to use PageRank in your queries, the first argument to the querier should be --pagerank, followed by the filepaths that lead to the title index (titles.txt), the document index (docs.txt), and the word index (words.txt) in that order. If you would not like to use PageRank, simply omit the argument --pagerank and keep the remaining three arguments in order.

Once the arguments have been passed to the querier, you can begin your search of the document. When prompted by the term "search", enter the word(s) that you would like to query. After, the results will be printed. To search with another query, simply enter the words(s) when prompted again. To end the query, enter :quit into the search prompt.

design & implementation

Indexer:

XML files are indexed in 4 main parts:

  1. pageTitleTable, which maps each unique page ID to the title of the corresponding page. an additional hash map, pageTitlesToIDS, is created here as well that maps page titles to their unique page IDs. This hash map is used in the later calculations for PageRank.
  2. idsToMaxFreqs(), which maps each unique page ID to the number of times the most commonly used word on the page appears on the page. This is achieved by going through pageTokenTable (a hash map of page IDs mapped to an array of the tokenized version of all the words that appear on that page), counting how many times each word appears on that page and finding the maximum value from there.
  3. pageRank(), which maps each page ID to its ranking (in the local variable rankMap). It uses helper methods that:
    1. check if one page links to another (accounting for the special cases) by checking pageLinksTable, which is created at the same time as the indexer itself that maps page IDs to a set of all the page IDs that this page links to
    2. calculate the weight of each pair of pages, accounting for the special cases (weight)
    3. calculate the distance between iterations of the ranking (distance)
  4. wordsToDocumentFrequencies(), which maps each word that appears in the corpus to a hashtable that maps each page ID to the number of times the word appears on that page. This is achieved by looping through each page, looking at each word on that page and either adding it to the big hashmap if it doesn't exist yet, or updating the value for that page on the small hashmap if it does.

Querier:

The querier runs its calculations by using the data structures that were created in the indexer. After the user has submitted a query, the input is appropriately tokenized and stored as the local variable "words". The helper method calcRank() is then called, which calculates the total page score for each page based on the input query words. The results are stored as the local variable idsToPageScores, which is a buffer of tuples, each containing the page IDs and their associated scores. Then, this buffer is converted into an array and sorted from highest to lowest score and the helper method printResults() prints the top 10 results (or fewer if less than 10 matching pages were found). If no results are found, "Could not find input on any pages. Try another query." is printed.

recommender

this project was written in java. given a CSV file of data, this project generates a decision tree and then makes predictions about new data based on that decision tree.

design & implementation

the decision tree is created using custom node and leaf classes, where each edge represents an attribute and each leaf represents a decision. the data is parsed from the csv format and then represented in this tree structure. given a target attribute, the tree is generated where each leaf is an option from that attribute and each node is one of the other non-target attributes. given a new point of data, the tree can also look up what the decision regarding that attribute should be, based on how similar it is to data that the tree was generated from.

model checking

this project was written in scala. this project is a model of how we might ensure that a program will always run safely: it's easy to test a small function, but it might not be possible to test all cases in a realistic system. this project models a program as a graph, and accordingly is able to check for safety (nothing bad happens) and liveness (something good happens).

design & implementation

there were two main approaches considered for the representation of the state machine: an adjacency matrix or an adjacency list, both of which are representations of graphs. i implemented both options, and found that the matrix implementation was more space efficient, while the list was more time efficient. i decided that in this instance, time was more valuable so i continued with that approach. additionally, the matrix had more limitations (since it had to use static arrays).

each node in the graph stored information about the current state of the program and the one-directional edges represented possible changes in the program at runtime.

each state machine, one initialized, can check to make sure a program always meets some condition (checkAlways()) or that it never meets some condition (checkNever()). specifically, only states that are reachable from the set startNode are checked.

maze

this project was written in pyret. this project is a "maze game" that is playable! this was one of my first ever cs projects, so it's got a special place in my heart. the objective is to help bleuno (our lovable lamp/bear) escape from outer space through the space ship. be careful though: make sure blueno doesn't run out of stamina... or else!

design & implementation

the built-in "reactors" in pyret provide much of the structure needed for the game. the reactor continuously calls my custom functions that draw and update the gamestate. the gamestate keeps track of the player position, player stamina, and gadget positions.

atoi

written in x86_64 assemply, this project implements the built-in C function atoi(), which converts a string argument to an integer.

design & implementation

atoi() goes through each ascii character in the input and loops through each one until it either reaches the end or finds a non-numerical character.

one of the biggest challenges of this project was that it had to be written entirely in machine code and not some other higher-level langauge, so i had to manage registers and accumulators in the system processor.

honestly, this project was not very long, but gave me a really good understanding of how registers and low-level processing work.
